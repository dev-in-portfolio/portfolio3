ORBIT, FLUX, MODES & SHOWTIME

What this lab is doing

TRANSIT 3D // PRESTIGE is simulating a simplified exoplanet transit: a planet orbits a star, and every time it crosses the stellar disk, the apparent brightness drops. The 3D viewport shows the orbit and alignment geometry. The light-curve panel on the right converts that geometry into a live flux value F(t), drawing the dip in real time as the planet passes in front of the star.

The lab is wired like a miniature product: there is a resilient boot path, a presentation mode, an AI advisor surface, and a small automation API (window.transitBot) so you can script a demo or have another app drive the camera.

Step-by-step: how to drive the demo

  1. Pick a system: use the KEPLER / TRAPPIST / HD 209 chips in the sidebar. Each preset adjusts star color, planet size, and default orbital speed to match a famous exoplanet configuration.
  2. Shape the transit: use Noise, Orbit Speed, Planet Size, and Impact (b) to change the geometry. As you drag, the 3D rig and the light curve update together.
  3. Flip visual modes: CALM is clean and analytic, EXTREME adds cinematic punch, and INSANE leans into scanlines, color punch, and audio texture. Modes are purely presentational; the underlying math stays stable.
  4. Ask the advisor: type into the AI panel on the right and switch tabs between COACH / CRITIC / IDEA. Even offline, the lab returns canned explanations so the surface never feels dead.
  5. Showtime: the SHOWTIME button in the header runs a tiny scripted sequence that flips systems, modes, and an AI line to act like a 20-second portfolio tour.

When you click ENTER LAB, this overlay fades away, but the layout and boot logic stay in place. Reloading the page always brings you back through this double-splash sequence so it feels like a deliberate cold open, not a raw demo.

ABOUT // TRANSIT 3D LAB

TRANSIT 3D // PRESTIGE is a single-file WebGL lab for exploring exoplanet transits. It is built to show real engineering craft: resilience, a11y, and just enough cinematic drama to impress a recruiter.

Use the system chips to jump between curated star systems, then tune noise, speed, orbit radius, and impact parameter to watch how the light curve responds. The right-hand panel stays in sync with the simulation.

Single-file WebGL lab WebGL + 2D canvas Accessible controls Keyboard-first UI Konami dev console State persistence

The app is structured like a tiny product: boot screen, live viewport, instrumentation, AI advisor, and a debug backdoor. It’s opinionated, but everything lives in one portable HTML file.

README // HOW TO DRIVE

Controls

  • OrbitControls: drag to rotate, scroll to zoom, right-drag to pan.
  • Noise / Speed / Radius / Impact sliders: update the transit geometry and sampling in real time.
  • PRESENT button: toggles presentation mode (fullscreen 3D focus; sidebar fades out).
  • AI tabs: switch between Coach / Critic / Idea advisors.

Persistence & State Hygiene

  • Key knobs (system, sliders, persona, theme, demo flag, visual mode) are saved to localStorage under a single namespaced key.
  • On load, we restore state, re-apply the theme, and call setSystem() so the viewport and HUD are consistent.
  • Helper persistState() is called only from user-driven paths (sliders, buttons, persona changes).

Automation Hook

A tiny automation surface is exposed as window.transitBot with helpers like setSystem(), togglePresentation(), setVisualMode() and sendRobotPrompt() for scripted demos.

Robots & 404

  • A sample robots.txt lives in the dev console for deployments.
  • Visiting a deployed /404 route (or using #404 in the URL) will trip a tiny “Lost in transit” page overlay.
